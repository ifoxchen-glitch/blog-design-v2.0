用真实长文压版式：单篇文章模板

2026年4月1日 作者：设计原型 前端 排版

目录

  • 为什么要压版
  • 设计令牌与 CSS
  • 间距与行宽
  • 代码块与行内代码
  • 列表与引用
  • 小结

空模板上的标题与段落往往看起来「刚好」；一旦换成真实长文、中英文混排、列表和代码，问题才会暴露：标题层级是否跳脱、代码块是否压迫正文、引用与图片说明是否像附属品。设计交付里用 2～3 篇完整样例压一次版式，能显著减少开发返工。

设计令牌与 CSS

本原型将 Figma 中的 Variables 思路映射为 css/tokens.css 中的自定义属性：颜色、字号阶梯、间距（8px 栅格）、圆角与正文最大宽度 --prose-max。深色模式通过 prefers-color-scheme: dark 切换同一套语义令牌，避免硬编码重复。 

/* 令牌消费示例 */
.prose {
  max-width: var(--prose-max);
  font-size: var(--text-body-lg);
  line-height: var(--leading-relaxed);
}

间距与行宽

中文长文可将最大行宽设在约 38～42rem（或约 35～45 全角字符的视觉感受），略宽于英文常见的 65ch，以兼顾标点与无空格分词。段间距应明显大于行内行高，否则读者难以扫读段落边界；本模板中段落下边距为 --space-4，二级标题上边距为 --space-6，形成稳定的垂直韵律。

「先定垂直韵律，再定组件。」同一套间距令牌应同时用于卡片内边距、列表页栅格间隙与 prose 区块，这样整站才会像同一作者写的，而不是页面拼装。

代码块与行内代码

行内代码使用等宽字体与浅底，避免与链接蓝色混淆；代码块单独设置水平滚动，防止撑破窄屏。下面是一段稍长的 TypeScript 片段，用于检查滚动条与行高。 

type Post = {
  title: string;
  slug: string;
  publishedAt: string;
  excerpt?: string;
  body: string;
  categories: string[];
  tags: string[];
};

async function loadPost(slug: string): Promise<Post | null> {
  const res = await fetch(`/api/posts/${encodeURIComponent(slug)}`);
  if (!res.ok) return null;
  return res.json() as Promise<Post>;
}

若文章内有多段短命令，也可用列表承载，避免连续多个单行 pre 块造成视觉碎裂： 

# 安装依赖（示例）
npm install
npm run dev

列表与引用

检查清单式内容在 prose 中的缩进与项目间距：

  • 首页：Hero、文章列表、页脚链接区是否对齐同一套水平 padding。
  • 列表页：卡片标题两行截断与摘要三行截断是否在不同宽度下仍整齐。
  • 单篇：目录在桌面 sticky 时是否与正文首行对齐；移动端目录是否改为文首折叠（本原型未做折叠，可迭代）。

有序步骤示例：

  1. 在 Figma 建立 Color / Space / Type 三套变量。
  2. 导出或手工维护与开发共用的 tokens 表（本仓库用 CSS 文件代替）。
  3. 用真实文章复制进 Frame，调整 H2 前间距与代码块上下间距。

小结

本页集中堆叠了标题、段落、列表、引用、多段代码与图片说明，用于验证 .prose 与全局令牌是否足够；若某元素显得「太紧」或「太散」，优先调整令牌而非单页魔法数，整站会一并受益。
